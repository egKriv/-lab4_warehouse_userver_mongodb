# Event-Driven архитектура системы управления складом (Вариант 18)

Документ описывает проект событийно-ориентированной архитектуры для складского REST API, построенного на фреймворке Userver с PostgreSQL и MongoDB. Реализация не входит в данную работу – представлено только проектное решение.

## 1. Анализ событий

Система содержит следующие значимые события, возникающие в результате выполнения команд.

| Команда (Command)       | Событие (Event)            | Описание события                                   |
| ----------------------- | -------------------------- | -------------------------------------------------- |
| `CreateUser`            | `UserCreated`              | Зарегистрирован новый пользователь                 |
| `CreateProduct`         | `ProductCreated`           | Добавлен новый товар                               |
| `UpdateProduct`         | `ProductUpdated`           | Изменены атрибуты товара (название, цена, sku)     |
| `CreateReceipt`         | `ReceiptCreated`           | Зафиксировано поступление товара                   |
| `WriteOff`              | `InventoryWriteOff`        | Выполнено списание товара                          |
| (производное)           | `InventoryStockChanged`    | Изменилось количество товара на складе (как следствие поступления или списания) |

Событие `InventoryStockChanged` может порождаться обработчиками двух предыдущих событий для обновления read-модели.

## 2. Компоненты архитектуры

### 2.1. Производители событий (Producers)

- **Основной API-сервис (inventory-api)** – после успешного выполнения команды публикует соответствующее событие в брокер сообщений.
- В будущем возможно подключение IoT-устройств, генерирующих события типа `StockCheck`, `ItemDamaged` и т.п.

### 2.2. Потребители событий (Consumers)

- **Notification Service** – отправляет уведомления (email, push) при важных операциях (списание, низкий остаток, регистрация пользователя). Подписывается на события `UserCreated`, `InventoryWriteOff`, `ReceiptCreated` (при необходимости).
- **Analytics Service** – собирает статистику по движению товаров, строит отчёты. Подписывается на все события, связанные с изменениями товаров и складских операций.
- **CQRS Projector (Read-model updater)** – слушает все события (`ProductCreated`, `ProductUpdated`, `ReceiptCreated`, `InventoryWriteOff`, `InventoryStockChanged`) и обновляет денормализованные read-таблицы для быстрых запросов.

### 2.3. Брокер сообщений

Выбран **RabbitMQ** по причинам:
- Гибкая маршрутизация (topic exchange), удобная для подписки на различные типы событий.
- Зрелая экосистема и поддержка гарантий доставки через AMQP.
- Достаточен для текущего масштаба, проще в эксплуатации, чем Kafka.

## 3. Поток событий (Event Flow)

1. Клиент вызывает команду (REST API) – например, `POST /api/inventory/write-off`.
2. Обработчик команды обновляет состояние в PostgreSQL (и при необходимости в MongoDB).
3. При успешном выполнении публикуется событие `InventoryWriteOff` в RabbitMQ.
4. Брокер маршрутизирует сообщение всем подписанным очередям на основе routing key (например, `event.InventoryWriteOff`).
5. Потребители получают сообщение и выполняют свою логику независимо:
   - Notification Service отправляет оповещение.
   - Analytics Service сохраняет запись в аналитической БД.
   - CQRS Projector обновляет read-модель.

## 4. Формат сообщений

Сообщения передаются в формате JSON. Общая структура (envelope):

```json
{
  "eventId": "uuid",
  "eventType": "InventoryWriteOff",
  "timestamp": "2026-05-24T19:00:00Z",
  "payload": {
    // поля, специфичные для события
  }
}
```

Поле eventId уникально и используется для обеспечения идемпотентности потребителей.

5. Маршрутизация и гарантии доставки
Используется topic exchange с ключами вида event.<EventType>.

Каждая очередь потребителя привязана к нужным ключам, что позволяет гибко подписываться на одно или несколько событий.

Гарантии доставки: at-least-once. Это означает, что сообщение может быть доставлено повторно в случае сбоя. Потребители обязаны быть идемпотентными – повторная обработка не должна приводить к дублированию действий.

Идемпотентность достигается сохранением eventId обработанных сообщений в локальной БД потребителя с проверкой уникальности перед выполнением логики.

6. Применение CQRS
В системе целесообразно разделить ответственность команд и запросов (CQRS):

Команды (write side) – CreateUser, CreateProduct, UpdateProduct, CreateReceipt, WriteOff.
Реализованы в текущем API с использованием PostgreSQL как основного хранилища. После выполнения команды публикуется событие.

Запросы (read side) – получение списка товаров, поиск по названию, остатки, история поступлений и списаний.
Могут обслуживаться отдельным сервисом или теми же endpoint'ами, но данные для них предоставляет read-модель, построенная асинхронно через обработку событий.

Read-модель содержит денормализованные представления, оптимизированные под конкретные запросы:

product_stock_summary – товар + текущий остаток + дата последнего изменения.

writeoff_history – история списаний с полями, готовыми к выдаче (без JOIN).

receipt_history – аналогично для поступлений.

CQRS Projector, слушая все события изменений, поддерживает эти таблицы в актуальном состоянии (eventual consistency). Благодаря этому запросы не затрагивают транзакционные таблицы PostgreSQL и выполняются максимально быстро.

7. Диаграмма потоков (упрощённо)
text
Client -> API (Command) -> PostgreSQL + MongoDB (write)
                          |
                          v
                      RabbitMQ (event)
                     /      |      \
                    /       |       \
         Notification   Analytics   CQRS Projector
                                        |
                                        v
                                   Read DB (PostgreSQL/MongoDB)
                                        ^
                                        |
                                Query Service (read endpoints)
8. Заключение
Предложенная архитектура обеспечивает слабую связанность компонентов, масштабируемость и возможность независимого развития потребителей. Внедрение брокера сообщений и CQRS позволит разделить нагрузку записи и чтения, повысить отказоустойчивость и упростить добавление новой функциональности (например, интеграция с системами бухгалтерии, службами доставки).
