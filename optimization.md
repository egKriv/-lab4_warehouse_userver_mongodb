### Индексы

- `idx_users_login` – для поиска по логину (авторизация).  
- `idx_products_name_trgm` – для быстрого поиска товаров по части названия (`%слово%`). Потребовалось расширение `pg_trgm`.  
- `idx_receipts_product_date` – для фильтрации по товару + сортировки по дате (вместо двух отдельных индексов).  
- Остальные индексы (по email, sku, created_by и т.д.) – стандартные, под типовые фильтры.


### Скорость

**1. Поиск пользователя (`login = 'user50'`)**  
С индексом: `Index Scan` за **0.059 мс**.  
Без индекса: `Seq Scan` за **0.039 мс**.  
На 100 записях индекс только замедлил поиск. Нужно большее число записей, но в системе управления складом даже 100 пользователей может не набраться

**2. Поиск товара по маске (`name ILIKE '%50%'`)**  
С обычным B-tree индексом: планировщик его проигнорировал, сделал `Seq Scan` за **0.114 мс** (99 строк отбросил).  
С GIN-индексом (pg_trgm): время упало до **0.036 мс**, хотя план всё ещё `Seq Scan` из‑за малого размера таблицы.

**3. Поступления по товару с сортировкой по дате**  
С отдельными индексами (`product_id` и `receipt_date`): план включал сортировку (`quicksort`), общее время **0.911 мс** для 1000 записей.  
С композитным индексом `(product_id, receipt_date DESC)`: время выполнения упало до **0.022 мс** – в 40 раз быстрее. Есть эффект даже на 1000 записей, а списаний может быть учтено еще больше



### Итог

Индексы работают. Композитный индекс для поступлений дал самый заметный выигрыш (с 0.9 мс до 0.02 мс на 1000 строк).
